## Context

Проект создается как backend для уже готового frontend dashboard-прототипа. UI показывает пять ключевых экранов: overview, tasks, mentors, messages и settings. Требования тестового задания задают не только стек, но и архитектурный стиль: все обращения к БД должны идти через repository layer, основная бизнес-логика через action / command classes, а внешние интеграции через service layer.

Laravel 11 уже создан в корне проекта. Для AI-assisted разработки в проект установлены Laravel Boost и OpenSpec. Для тестов подключен Pest. Среда пока не настроена под реальный PostgreSQL-инстанс, поэтому миграции и тесты будут зависеть от последующей конфигурации БД.

## Goals / Non-Goals

**Goals:**
- Покрыть API почти для всего предоставленного frontend-а.
- Показать последовательную архитектуру `Controller -> Action -> Repository / Service`.
- Спроектировать БД так, чтобы overview собирался из реальных доменных сущностей, а не из вручную склеенных массивов.
- Формализовать обязательный набор тестов на ключевые API-флоу и архитектурные ограничения.
- Сохранить объем реализации реалистичным для тестового задания.

**Non-Goals:**
- Не реализовывать realtime chat, websocket и online presence.
- Не строить полноценную админ-панель.
- Не внедрять тяжелые внешние интеграции вроде email, sms или push.
- Не реализовывать сложные auth-flow, если они прямо не нужны для демо-режима.

## Decisions

### 1. Backend будет организован вокруг capability-ориентированного API
Почему:
- frontend уже разделен на пять крупных функциональных экранов;
- такой разрез удобно маппится и на эндпоинты, и на action-классы, и на тесты.

Альтернативы:
- Делить только по сущностям (`users`, `tasks`, `mentors`) было бы проще на уровне CRUD, но хуже объясняет `overview`, который является агрегатом нескольких доменов.

### 2. База данных будет опираться на нормализованные сущности
Решение:
- `users`, `categories`, `mentors`, `tasks`, `task_steps`, `task_user`, `mentor_followers`, `reviews`, `conversations`, `conversation_participants`, `messages`, `user_settings`.

Почему:
- этого достаточно, чтобы покрыть все экраны;
- модель остается понятной;
- можно честно агрегировать overview из БД.

Альтернативы:
- хранить агрегаты напрямую в одной таблице было бы быстрее, но выглядело бы как mock-backend, а не как продуманная архитектура.

### 3. Repository Pattern используется только для БД-операций
Решение:
- каждый домен получает интерфейс репозитория и Eloquent-реализацию;
- контроллеры не используют Eloquent напрямую;
- action-классы получают данные только через repositories.

Почему:
- это соответствует формулировке задания;
- архитектурно проще контролировать зависимости и тестировать action-классы.

Альтернативы:
- использовать Eloquent прямо в контроллерах быстрее, но это нарушит главный architectural requirement тестового.

### 4. Action / Command Pattern будет оформлен как один класс на один use case
Примеры:
- `GetOverviewAction`
- `ListTasksAction`
- `GetTaskDetailsAction`
- `ListMentorsAction`
- `FollowMentorAction`
- `ListConversationsAction`
- `SendMessageAction`
- `UpdateSettingsAction`

Почему:
- так проще показать, где именно находится бизнес-логика;
- контроллеры останутся тонкими;
- тесты можно писать на уровне use case.

Альтернативы:
- сервисы “на все случаи” ухудшат читаемость и размоют ответственность.

### 5. Service Layer будет продемонстрирован на внешнем avatar provider
Решение:
- ввести `AvatarServiceInterface` и `DiceBearAvatarService`;
- генерация/резолв avatar URL для пользователей и менторов будет проходить через service layer.

Почему:
- frontend уже использует внешние avatar URL;
- это реальная внешняя интеграция без избыточной сложности;
- хорошо закрывает требование `Tashqi servislarga integratsiyalar`.

Альтернативы:
- добавлять Twilio, Pusher или email provider для тестового избыточно.

### 6. Тестирование будет строиться вокруг Pest feature tests + architecture tests
Решение:
- feature tests на overview, tasks, mentors, conversations, settings;
- отдельные тесты на follow/unfollow и отправку сообщения;
- architecture tests на запрет прямого доступа контроллеров к Eloquent и на ожидаемую структуру action/repository/service.

Почему:
- это напрямую усиливает качество тестового;
- проверяющий увидит не только API, но и контролируемую архитектуру.

Альтернативы:
- ограничиться только happy-path API tests можно, но это слабее показывает инженерную зрелость.

## Risks / Trade-offs

- [PostgreSQL еще не настроен локально] -> Временно использовать правильный `pgsql`-конфиг в `.env`, а реальные миграции и тесты запускать после ввода рабочих кредов или поднятия локальной БД.
- [Отсутствие явного auth-flow во frontend] -> Для MVP разрешить demo-user контекст или минимальный auth слой без усложнения продуктовой модели.
- [Overview агрегирует много данных] -> Выделить отдельный action и репозиторий-агрегатор, чтобы не размазывать сборку ответа по контроллеру.
- [Message-модуль может разрастись] -> Ограничить scope только inbox, messages list и send message без realtime.
- [Repository Pattern можно перегрузить абстракциями] -> Держать интерфейсы прикладными и не выносить туда лишние generic-методы.

## Migration Plan

1. Настроить окружение Laravel под PostgreSQL.
2. Создать миграции и связи для всех доменных таблиц.
3. Подготовить сиды с demo-данными под UI.
4. Реализовать repositories.
5. Реализовать action-классы и services.
6. Подключить API routes, Form Requests и Resources.
7. Написать feature и architecture tests.
8. Прогнать тесты и финальную валидацию change.

Rollback:
- откат миграций;
- удаление новых маршрутов и классов;
- откат change до archive only после финального завершения.
